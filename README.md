# Talk-to-Data CLI

Ask questions about your data in plain English. The application translates your question into SQL using a large language model, runs it against DuckDB, and opens an interactive chart in your browser — with a built-in execution plan viewer that explains what DuckDB did internally.

```
User: "Who scored the most total points in NBA history?"
  → SQL generated by Groq (llama-3.3-70B)
  → Executed by DuckDB (in-memory, vectorized, columnar)
  → Bar chart opens in browser with DuckDB execution plan panel
```

## Getting a Groq API Key

The application uses the Groq API (free tier) to convert natural language questions into SQL.

1. Go to [https://console.groq.com](https://console.groq.com)
2. Sign up for a free account (no credit card required)
3. Navigate to **API Keys** in the left sidebar
4. Click **Create API Key**
5. Copy the key — it starts with `gsk_`

Free tier limits: **14,400 requests per day** — more than enough for normal use.

## Setting the API Key

Choose one of the following methods. The `.env` file method is the most convenient.

### .env file (recommended)

Create a file named `.env` in the project directory (same folder as `main.py`):

```
GROQ_API_KEY=gsk_your_key_here
```

The application loads this automatically on startup via `python-dotenv`. No other configuration needed.


## Dataset Setup

The repository includes a demonstration dataset in the `data/` directory. No additional download is required.

### Included Dataset — NBA Basketball Statistics

Three CSV files covering NBA player biographical and statistical data:

| File | Table Name | Rows | Columns | Description |
|---|---|---|---|---|
| `data/player_data.csv` | `player_data` | 4,550 | 8 | Player biographical info: career years, position, height, weight, birth date, college |
| `data/Players.csv` | `players` | 3,922 | 8 | Alternate player roster: height, weight, college, birth year, birth city/state |
| `data/Seasons_Stats.csv` | `seasons_stats` | 24,691 | 53 | Per-season statistics: points, assists, rebounds, shooting percentages, advanced metrics (PER, WS, BPM, VORP) |

**Total: 33,163 rows across 3 tables.**

The dataset spans NBA seasons from approximately 1950 to 2017 and is sourced from Basketball-Reference.com.

### Using Your Own Data

You can load any CSV or Parquet file. When the application starts, choose option `2` at the file loading prompt and enter a file path or folder path.

**Supported formats:**
- `.csv` — column types are auto-detected (delimiter, encoding, and BOM are handled automatically)
- `.parquet` — schema is preserved exactly as-is

**You can load:**
- A single file: `C:\data\sales.csv` or `/home/user/data/sales.csv`
- An entire folder: `C:\data\my_dataset\` (all CSV and Parquet files inside are loaded)
- Files already placed manually in the `data/` folder (choose option `1`)

## Session Walkthrough

### Step 1 — Load Data Files

```
══════════════════════════════════════════════════════════════════
  STEP 1 — Load Data Files
══════════════════════════════════════════════════════════════════

  Files in data/ (3):
    • player_data.csv
    • Players.csv
    • Seasons_Stats.csv

  Options:
    [1] Use files already in the data/ folder
    [2] Enter a file or folder path to load

  Enter 1 or 2: 1
```

Enter `1` to use the included NBA dataset, or `2` to load your own files.

After loading, DuckDB registers each file as an in-memory table. Date columns are automatically detected and converted so time-based queries work correctly.

### Step 2 — Build Schema

```
══════════════════════════════════════════════════════════════════
  STEP 2 — Building Schema
══════════════════════════════════════════════════════════════════

  3 table(s) detected → will use full schema strategy

  Phase 1: scanning table structure...
    Scanning player_data...  8 cols  [0.1s]
    Scanning players...      8 cols  [0.1s]
    Scanning seasons_stats...  53 cols  [0.2s]

  Phase 2: generating column descriptions with Groq...
  (Runs once per table, results cached — improves SQL accuracy)

    player_data...  ✅  8/8 cols described  [1.2s]
    players...      ✅  8/8 cols described  [1.1s]
    seasons_stats...  ✅  53/53 cols described  [2.3s]

  ✅  Schema cached to registry/schema_cache.json
```

**Phase 1** (fast, always runs) scans column names and types from DuckDB's catalog.

**Phase 2** (runs once, then cached) calls the Groq API to generate a plain-English description for every column. This makes SQL generation significantly more accurate because the LLM understands what columns like `PER`, `VORP`, and `WS/48` mean. On subsequent runs, Phase 2 is skipped and the cached descriptions are loaded instantly.

### Step 3 — Ask Questions

```
══════════════════════════════════════════════════════════════════
  STEP 3 — Ask Questions
══════════════════════════════════════════════════════════════════
  Ask anything about your data in plain English.
  Commands:  schema | charts | history | quit

──────────────────────────────────────────────────────────────────
  Your question: Who scored the most total points in NBA history?

  Schema mode: full  (3 tables total, 3 sent to model)
  Generating SQL...
  Model: llama-3.3-70b-versatile

  Generated SQL:
  ────────────────────────────────────────────────────────────────
    SELECT "Player", SUM("PTS") AS total_points
    FROM seasons_stats
    GROUP BY "Player"
    ORDER BY total_points DESC
    LIMIT 10
  ────────────────────────────────────────────────────────────────

  Result: 10 rows × 2 columns

   Player              total_points
   Kareem Abdul-Jabbar        38387
   Karl Malone                36928
   ...

  Auto-detected chart: bar
  Chart type [bar/line/pie/scatter/table] (ENTER = keep 'bar'):

  ✅  Chart saved → charts/who_scored_the_most_total_points_in_nb.html
  ✅  Opened in your browser ✨
  Tip: Click '🔍 Explain Query' in the chart to see the DuckDB execution plan
```

A chart opens in your browser. Click **Explain Query** in the top-right toolbar to open the DuckDB execution plan panel, which shows each operator DuckDB used (Hash Join, Sequential Scan, Hash Group-By, etc.) with plain-English explanations.

│   ├── player_data.csv      4,550 rows — player biographical info

│

### How Data Flows

```
CSV file → data/
    ↓
db.py: read_csv_auto() or pandas fallback → DuckDB in-memory TABLE
    ↓
schema.py: information_schema queries → column names + types + row counts
    ↓  (optional, once)
llm.py: Groq API → column descriptions (cached to schema_cache.json)
    ↓  (each question)
schema.py: build schema string (full or TF-IDF retrieved)
    ↓
llm.py: Groq API (llama-3.3-70B) → SQL string
    ↓
db.py: DuckDB execute → pandas DataFrame
db.py: EXPLAIN → plan text
    ↓
charts.py: Plotly figures + parsed plan → self-contained HTML
    ↓
Browser opens HTML file
```

## Troubleshooting

### "GROQ_API_KEY not set"
The application can scan tables and display schema without the key, but cannot generate SQL. Set the key using any method described in [Setting the API Key](#setting-the-api-key). Verify it is loaded by checking the startup banner — it shows the first 12 characters of the key if detected.

### "SQL execution failed"
The LLM occasionally generates SQL with a wrong column name or incorrect JOIN logic. When this happens:
- The error message is displayed along with the available table and column names.
- Try rephrasing your question using the exact column names shown.
- Example: instead of "three point percentage", say "3P%" or "3-point percentage".

### "No CSV/Parquet files found"
Make sure the `data/` directory exists and contains at least one `.csv` or `.parquet` file. If you moved the data files, use option `2` at the file loading prompt to re-specify the path.

### Charts do not open in the browser
The HTML files are saved in the `charts/` folder. You can open them manually:
- Windows: double-click the `.html` file in File Explorer
- macOS/Linux: `open charts/filename.html` or `xdg-open charts/filename.html`

The chart files are self-contained — no internet connection is needed to view them (Plotly is bundled via CDN script tag; a one-time CDN load is required for the Plotly library).

### Schema cache is stale
If you add or remove files from `data/` between runs, the application detects that the cached table list no longer matches and automatically rebuilds the schema. You can also delete `registry/schema_cache.json` to force a full rebuild.

### "All Groq models failed"
- Check that your API key is correct (starts with `gsk_`).
- Check Groq service status at [https://status.groq.com](https://status.groq.com).
- The application tries `llama-3.3-70b-versatile` first, then falls back to `llama-3.1-8b-instant` automatically before giving up.

### UnicodeEncodeError on Windows
The application reconfigures `stdout` to UTF-8 automatically on startup. If you still see encoding errors, run:
```cmd
chcp 65001
python main.py
```

### CSV file loads as a single column
This usually means the file has a non-comma delimiter or a BOM. The application handles this automatically through its three-attempt strategy (DuckDB auto-detect → pandas utf-8-sig → explicit delimiter scan). If all three fail, the file format may be unsupported — try opening it in a text editor to inspect the delimiter and encoding, then convert it to standard UTF-8 CSV.

## Schema Strategy

| Schema size | Strategy | Behavior |
|---|---|---|
| ≤ 50 tables | Full schema | All table and column info sent to the LLM in one prompt |
| > 50 tables | TF-IDF retrieval | Top 5 most relevant tables selected per question using cosine similarity |

The threshold is defined in `schema.py` as `RETRIEVAL_THRESHOLD = 50`. For the included NBA dataset (3 tables), the full schema is always used.
